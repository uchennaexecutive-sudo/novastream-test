01

The Big Picture

What Nova Stream is, how it’s built, and what happens when you press Play

🚀
Opening Hook

You just pressed Play on a movie. In the next 3 seconds, 4 completely different systems fired up — a React UI, a Tauri shell, a Rust engine, and a local streaming server. Here’s how they all work together.

What is Nova Stream?

Nova Stream is a premium streaming desktop application for Windows and macOS. It looks and feels like Netflix, but runs as a native app installed on your computer. You browse, you press Play, and a video starts — no browser tab, no Flash plugin, no web dependency.

🎬

Movies

Browse, search, and watch feature films. Streams are resolved on the fly by a local server running on your machine, not a remote CDN.

📺

Series

Full episode navigation, season switching, and a resume system that remembers exactly where you left off — per episode, not just per show.

⛩️

Anime

Powered by AniList with its own separate playback engine and scraping logic that’s isolated from the movie/series path.

✨

Animation

Uses the same stream resolver as movies but with a tuned provider preference so it doesn’t compete with movie-specific fast paths.

💻
It’s a real app, not a website

Nova Stream can access your file system, run native code, download updates silently, and make network requests that a web browser would refuse. This is the key difference — and it changes everything about how it’s built.

Three Layers of the App

Think of it like a spaceship. The cockpit (React) is what the pilot touches. Mission control (Tauri) is the bridge that makes the ship real. The engines (Rust) do the propulsion no human could do by hand.

🔩
Why Rust? The honest answer.

Some things — like grabbing a video stream that blocks browsers, or downloading a 50MB update without freezing the UI — need to run in a native process, not inside a web tab. Rust is the language that makes that possible while keeping the app safe and fast. CORS restrictions that would stop a web app cold are completely irrelevant to Rust.

The Movie Engine

How pressing Play launches a 3-stage intelligence operation to find your stream

The Talent Agency Problem

Clicking Play on a movie does not just open a video file. It launches a 3-stage intelligence operation — a local server, a Rust validator, and a cascade of provider queries — all completing in under 8 seconds.

Think of it like a talent agency casting an actor for a role. They call their top client first. If unavailable, they call a fast shortlist. If still nothing, they go through the full roster. The first one that says "yes" gets the part.

★

Stage 1 — Primary

One provider. One call. vixsrc is tried alone. Most popular titles resolve here instantly.

Timeout: ~8s

▶▶

Stage 2 — Fast Set

Four providers tried quickly in parallel: vixsrc, moviesmod, 4khdhub, moviesdrive.

Timeout: 18s

📄

Stage 3 — Full Fallback

All 12 providers tried including showbox, vidsrc, soapertv and more.

Timeout: 75s

⚡
Why 3 stages?

Speed vs. coverage. Most titles work on stage 1 or 2 in under 5 seconds. Stage 3 is the nuclear option for obscure titles that need all 12 providers tried. The cascade means common titles stay fast while rare titles still get found.

vixsrc Preferred primary — most reliable stream quality
moviesmod Fast set member — strong coverage for recent releases
moviesdrive Fast set member — good for international titles
4khdhub Fast set member — deprioritised behind the above three
showbox / vidsrc Full fallback only — broad but slower to resolve

The Subtitle Parallel Track

While the stream is loading, Rust simultaneously calls the Wyzie subtitle service. Subtitles do not block playback — they arrive alongside the video and snap into the player when ready. Movies use subdl/opensubtitles sources; series use gestdown; animation uses jimaku and animetosho.

Reading the Code: movieStreams.js

This file is the entire bridge between React and the Rust stream engine. It is intentionally thin — 41 lines with no business logic. Every important decision lives in Rust.

movieStreams.js import { invoke } from '@tauri-apps/api/core' async function fetchResolvedStreams( tmdbId, contentType, season, episode, imdbId, options = {} ) { return invoke( 'fetch_movie_resolver_streams', { payload: { tmdbId: String(tmdbId), contentType, season: season ?? null, episode: episode ?? null, imdbId: imdbId || null, forceRefresh: Boolean(options.forceRefresh), excludeUrls: Array.isArray(options.excludeUrls) ? options.excludeUrls : [], excludeProviders: Array.isArray(options.excludeProviders) ? options.excludeProviders : [], }} ) } export async function getMovieStreams( tmdbId, imdbId = null, options = {} ) { return fetchResolvedStreams( tmdbId, 'movie', null, null, imdbId, options ) } export async function getSeriesStreams( tmdbId, season, episode, imdbId = null, options = {} ) { return fetchResolvedStreams( tmdbId, 'series', season, episode, imdbId, options ) }
In plain English
Line 1 — Import invoke — the single function that opens a portal from React into Rust.
Lines 3-21 — Build the complete request object with every detail Rust needs: what content, what ID system, which season and episode if applicable.
Line 8 — invoke() is the exact moment control passes to Rust. React waits here, paused, until Rust is done.
Lines 15-20 — excludeUrls and excludeProviders let the player skip streams that already failed — enabling automatic fallback.
Lines 23-29 — getMovieStreams is a thin wrapper. Movies have no season or episode, so those are hard-coded null.
Lines 31-39 — getSeriesStreams is the same idea but adds season and episode numbers to the payload.
🔍
Notice how thin this file is

Just 41 lines — and zero business logic. React does not know which providers exist, how validation works, or what a HLS manifest is. It just packages the request and hands it to Rust via invoke(). All the complexity lives in Rust, where it belongs.

Reading the Code: The Rust Provider Strategy

The 3-stage cascade is defined as six constants at the top of main.rs. Changing provider order or timeouts is a one-line edit — the rest of the engine adapts automatically.

main.rs (lines 55-66) const NUVIO_PRIMARY_PROVIDER_MOVIE: &str = "vixsrc"; const NUVIO_FAST_PROVIDER_SET_MOVIE: &str = "vixsrc,moviesmod, 4khdhub,moviesdrive"; const NUVIO_FULL_PROVIDER_SET_MOVIE: &str = "vixsrc,moviesmod, 4khdhub,moviesdrive,moviebox, hdhub4u,topmovies,soapertv, vidzee,mp4hydra,vidsrc,showbox"; const NUVIO_FAST_STAGE_TIMEOUT_SECS: u64 = 18; const NUVIO_STREAM_FETCH_TIMEOUT_SECS: u64 = 75;
In plain English
PRIMARY — Vixsrc is trusted above all others. It gets a solo attempt before any other provider is involved.
FAST SET — Four providers tried together in the second stage. They are the same reliable names, plus two strong alternates, given 18 seconds total.
FULL SET — All 12 providers, including slower and more obscure ones like showbox and mp4hydra. The nuclear option.
18 seconds — The fast stage deadline. Short enough to stay snappy; long enough for a slow provider to reply.
75 seconds — The full fallback budget. Patience matters here — some providers are slow starters but do have the stream.

Why does Rust do this — not React?

🚫
Browsers block cross-origin requests

A web page calling a provider URL directly triggers a CORS error. The browser refuses to fetch it. The request never leaves the tab.

✅
Rust runs outside the browser sandbox

Rust is a native process. It is not a web page, so CORS rules do not apply. It can make HTTP requests to any host, carry custom headers, and handle manifests and segments that a browser iframe would refuse.

🔄
Rust proxies the entire HLS pipeline

After finding a stream, Rust rewrites the manifest so every segment and encryption key request passes through it. The native video element thinks it is talking to a local server — no CORS, no auth issues.

🔒
The CORS Problem in one sentence

Browsers block cross-origin requests as a security feature. Rust bypasses this entirely — it is a native process, not a web page, so it can fetch from anywhere. This is why the entire stream resolver lives in Rust and not in a .js file.

What is the Nuvio Sidecar?

The sidecar is a small local server — embedded inside the Nova Stream install — that handles the actual provider queries. Rust manages its lifecycle: starts it on first play, keeps it alive, and restarts it if it crashes.

1
Embedded in the app

The Nuvio runtime is bundled inside the Nova Stream installer. No separate install required.

2
Starts silently

On Windows it runs hidden in the background, logging to %TEMP%\nova-stream-nuvio-sidecar.log.

3
Self-healing

If the extracted runtime is deleted, the app recreates it automatically on the next launch.

The Detective Agencies

Think of the two providers as rival detective agencies assigned to the same case. Gogoanime gets first crack at every episode — they are fast and handle most titles cleanly. AnimePahe is on standby: if Gogo reports "case unsolvable," Pahe takes over from scratch with their own methods.

The Scoring System

When Gogoanime searches for "Naruto Shippuden," it gets back dozens of results — dubbed versions, spin-offs, wrong seasons. The scoreMatch() function in resolveAnimeStreams.js scores every result so the closest one wins.

resolveAnimeStreams.js function scoreMatch(match, titles = []) { const baseTitle = normalizeText( match?.title || match?.matchedTitle || '' ) if (!baseTitle) return -1 const descriptorTokens = extractDescriptorTokens(titles) const descriptorOverlap = countTokenOverlap( baseTitle, descriptorTokens ) const requestsDub = titles.some( (t) => /\bdub\b/i.test(String(t || '')) ) const isDubMatch = /\bdub\b/i.test(match?.title || '') || /-dub$/i.test(match?.animeId || '') let best = -1 for (const rawTitle of titles) { const title = normalizeText(rawTitle) if (!title) continue let score = 0 if (baseTitle === title) score += 200 else if (baseTitle.startsWith(title)) score += 120 else if (baseTitle.includes(title)) score += 80 if (descriptorTokens.length) { score += descriptorOverlap * 50 if (descriptorOverlap === 0) score -= 120 } if (requestsDub) { if (isDubMatch) score += 40 else score -= 40 } else { if (isDubMatch) score -= 60 } if (score > best) best = score } return best }
Plain English
normalizeText() — strips punctuation and lowercases both titles so "Attack on Titan!" and "attack on titan" compare as equal.
descriptorTokens — keywords that distinguish a season or arc, like "Shippuden," "Final Season," or "Part 2." These must appear in the result too.
score += 200 — exact title match. The gold standard. A perfect hit against any title in the AniList titles array wins outright.
score += 120 — the result title starts with our search term. Common when a provider appends extra words like "Season 2."
score += 80 — our term appears somewhere inside the result. Weaker signal but still valid.
descriptorOverlap * 50 — each matching season/arc keyword adds 50 points. No overlap at all subtracts 120 — the result is probably a different arc entirely.
isDubMatch penalty -60 — if the user did not ask for a dub and this result is a dub version, subtract 60 points. Subtitles are the default preference.
return best — the function runs across every title variant (English, romaji, synonyms) and returns the highest score achieved.
🎯
The Scoring System Is Critical

"Naruto Shippuden" returns dozens of results on Gogoanime — dubbed versions, movies, wrong arcs. Without scoring, a dubbed version or the wrong season could win silently. The point system ensures the closest title with the right audio track always comes out on top.

Score Breakdown Example
+200 Exact title match — "Naruto: Shippuden" === "Naruto: Shippuden"
+120 Starts-with match — result is "Naruto Shippuden Season 2," request is "Naruto Shippuden"
+50 Per matching descriptor token — "Shippuden" appears in both request and result
-60 Dub penalty — result is a dubbed version, but user requested subtitles
-120 Zero descriptor overlap — result has no "Shippuden," likely a different arc entirely

Session-Aware Fetching

AnimePahe is a provider that requires a live browser session — it rejects requests that arrive without the right cookies. Rust acts as a persistent browser, maintaining that session across every request for the same title.

If a cookie expires mid-stream, the session manager detects the error, clears the stale session, and silently retries with a fresh one — up to three attempts.

animepahe.js async function fetchProviderTextWithSession( url, headers = {}, method = 'GET', body = null, animeId = '', preferredSessionId = null ) { let lastError = null for (let attempt = 0; attempt < 3; attempt++) { const useFreshSession = attempt > 0 const sessionId = useFreshSession ? null : resolveProviderSession(animeId, preferredSessionId) try { if (useFreshSession) clearProviderSession(animeId) const result = await invoke('fetch_anime_text_with_session', { url, headers: { ...BROWSER_HEADERS, ...headers }, method, body, sessionId, }) const text = typeof result?.text === 'string' ? result.text : '' const nextSessionId = rememberProviderSession( animeId, result?.sessionId || sessionId ) return { text, sessionId: nextSessionId } } catch (error) { lastError = error if (!isRetryableSessionError(error) || attempt >= 2) throw error } } throw lastError || new Error('AnimePahe session fetch failed') }
Plain English
animeId — the title being fetched. Used as the key to look up any existing session cookie for this anime.
attempt loop (0–2) — tries up to three times. First attempt reuses the cached session; later attempts start fresh if it expired.
useFreshSession — on retry, this is true. The old session is cleared so a brand-new cookie is obtained instead.
invoke('fetch_anime_text_with_session') — calls into Rust, which makes the actual HTTP request carrying the session cookie as a browser would.
BROWSER_HEADERS — standard browser headers (User-Agent, Accept, Referer). Without these, the server would reject the request as a bot.
rememberProviderSession() — saves the session ID returned by the server so the next request reuses it, avoiding repeated session warmup.
isRetryableSessionError() — checks if the error is a session expiry (worth retrying) versus a hard failure (not worth retrying).
🦀
Rust as the Browser

Rust's reqwest client maintains a per-provider cookie jar keyed on anime ID. It behaves like a real browser session, so AnimePahe servers accept its requests.

🔁
Three-Attempt Retry

Attempt 1 uses the cached session. If it fails with a retryable error, attempt 2 clears the cache and starts a fresh session. Attempt 3 is the final try before throwing the error up.

💾
Session Caching

After a successful fetch, the new session ID is saved to an in-memory map keyed by anime ID. The next episode request for the same show reuses the warm session instantly.

The Write-Through Pattern

Every time you bookmark a title, Nova Stream uses a write-through pattern — the local save happens in the same frame as your click. The cloud write happens later, quietly, without blocking the UI.

supabase.js export const addToWatchlist = async (item) => { const list = readStore(WATCHLIST_KEY) const entry = { id: item.tmdb_id, tmdb_id: item.tmdb_id, media_type: item.media_type, title: item.title, poster_path: item.poster_path, added_at: new Date().toISOString(), } const idx = list.findIndex(i => i.tmdb_id === item.tmdb_id) if (idx >= 0) list[idx] = entry else list.unshift(entry) writeStore(WATCHLIST_KEY, list)   // Write-through to Supabase const userId = await getUserId() if (userId) { supabase.from('watchlist') .upsert({ user_id: userId, ... }, { onConflict: 'user_id,tmdb_id' }) .then(({ error }) => { if (error) console.warn('[watchlist] sync write failed') }) } }
Plain English
readStore(WATCHLIST_KEY) — pull the current watchlist from localStorage into memory.
Build the entry — create a clean object with title, media type, poster, and a timestamp.
findIndex check — if this title is already in the list, update it in place rather than duplicating it.
unshift(entry) — if it's new, add it to the very front of the list (newest first).
writeStore(...) — save back to localStorage immediately. This is synchronous — done in the same frame.
getUserId() — check if anyone is signed in. Guests skip the cloud step entirely.
upsert(...) — write to Supabase's watchlist table. onConflict means: if (user_id, tmdb_id) already exists, update it instead of erroring.
.then(...) — this callback runs later, in the background. The local save already happened frames ago.
if (error) console.warn — if the cloud write fails, log it quietly. The local copy is safe. Nothing breaks for the user.

Timeline: What happens when you click the bookmark

0 ms — User clicks bookmark

The UI responds instantly. The bookmark icon flips to filled.

0 ms — localStorage write (synchronous)

writeStore(WATCHLIST_KEY, list) — the data is safely on disk before the next line runs.

~5–50 ms — Supabase upsert (async, fire-and-forget)

The network request is kicked off. The UI doesn't wait. Fire-and-forget means a success or failure here has no effect on the UI.

Background — Cloud confirms (or silently fails)

If Supabase confirms: both notepads agree. If it fails: the local notepad is untouched. Either way, the user already moved on.

Fire and forget

The local save is synchronous — it happens in the same frame as the click. The cloud write is asynchronous — the UI never waits for it.

The Sync-Down Merge

When you sign in — or when the app starts and finds an existing session — it calls syncFromCloud(). This pulls your cloud data down and merges it carefully with whatever is already on the device.

supabase.js export const syncFromCloud = async () => { const userId = await getUserId() if (!userId) return   try { const { data: cloudWatchlist, error: wErr } = await supabase .from('watchlist') .select('*') .eq('user_id', userId) .order('added_at', { ascending: false })   if (!wErr && cloudWatchlist?.length) { const local = readStore(WATCHLIST_KEY) const cloudIds = new Set(cloudWatchlist.map(i => i.tmdb_id)) const localOnly = local.filter(i => !cloudIds.has(i.tmdb_id)) const merged = [ ...cloudWatchlist.map(i => ({ ...i, id: i.tmdb_id })), ...localOnly, ].sort((a, b) => new Date(b.added_at) - new Date(a.added_at)) writeStore(WATCHLIST_KEY, merged) } } catch (err) { console.warn('[sync] Cloud sync failed, using local data') } }
Plain English
getUserId() — is anyone signed in? If not, return immediately — guests are localStorage-only, no cloud fetch needed.
supabase.from('watchlist').select('*').eq('user_id', userId) — fetch all this user's watchlist entries from the cloud, newest first.
if (!wErr && cloudWatchlist?.length) — only proceed if the fetch succeeded and actually returned rows.
readStore(WATCHLIST_KEY) — read the current local list so we can compare it to the cloud version.
new Set(cloudWatchlist.map(...)) — build a fast-lookup Set of cloud IDs. Checking "is this ID in the Set?" is O(1) — instant, regardless of list size.
localOnly — items that exist on this device but haven't synced yet. We keep these; no data is thrown away.
merged = [...cloud, ...localOnly] — cloud items come first (cloud wins conflicts), then local-only items are appended.
.sort(...) — sort the entire merged result by date, so the newest bookmarks always appear at the top.
writeStore(...) — save the merged result back to localStorage. Local and cloud are now in agreement.
catch (err) — if the network fails entirely, log a warning and move on. Local data is never touched on a failed sync.

What the database looks like

Four tables back the sync layer, all protected by Row Level Security (RLS) so users can only access their own rows.

profiles avatar_style, avatar_seed, username, theme, preferences (JSONB) — one row per user
watchlist user_id, tmdb_id, media_type, title, poster_path, added_at
watch_history same as watchlist + season, episode, progress_seconds, watched_at
watch_progress content_id, season, episode, progress_seconds, updated_at, user_id — keyed on content × season × episode
🔀
Cloud wins, but nobody loses

If you added something locally before signing in, it stays in your list — it just gets sorted by date. No data is ever discarded during a sync.

Module 5 Summary

1
Two notepads, one truth

localStorage is always the first write target — instant, offline-capable. Supabase is a background mirror that makes data available everywhere.

2
Write-through = fast UI, resilient data

Every local action fire-and-forgets a cloud write. If the network is down, the local copy is already safe.

3
Sync-down on sign-in = cloud wins, nothing lost

Cloud data wins conflicts, but local-only items are kept and sorted into the merged result. No data is ever discarded.

4
Account is optional — always was

The onboarding flow includes a Skip option. Row Level Security (RLS) ensures cloud data is locked to each user's ID.
