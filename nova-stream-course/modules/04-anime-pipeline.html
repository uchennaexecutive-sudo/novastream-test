The Anime Pipeline

Why anime needs its own system — and how two detective agencies find your stream

Why Anime Needs Its Own Pipeline

Movies have a universal ID system — every platform agrees on IMDb IDs. Anime has no such luxury. Three separate databases each think they own the truth — and none of them agree with each other.

Why a Separate Pipeline?

Movies have IMDb IDs on every platform. Anime identities are fragmented across databases that do not talk to each other. A shared movie pipeline would never find the right episode — so anime gets its own dedicated resolver stack.

The Detective Agencies

Think of the two providers as rival detective agencies assigned to the same case. Gogoanime gets first crack at every episode — they are fast and handle most titles cleanly. AnimePahe is on standby: if Gogo reports "case unsolvable," Pahe takes over from scratch with their own methods.

Crucially, every new episode resets the assignment — both agencies start fresh, so a past AnimePahe win does not lock future episodes out of Gogoanime.

The Scoring System

When Gogoanime searches for "Naruto Shippuden," it gets back dozens of results — dubbed versions, spin-offs, wrong seasons. The scoreMatch() function in resolveAnimeStreams.js scores every result so the closest one wins.

resolveAnimeStreams.js function scoreMatch(match, titles = []) { const baseTitle = normalizeText( match?.title || match?.matchedTitle || '' ) if (!baseTitle) return -1 const descriptorTokens = extractDescriptorTokens(titles) const descriptorOverlap = countTokenOverlap( baseTitle, descriptorTokens ) const requestsDub = titles.some( (t) => /\bdub\b/i.test(String(t || '')) ) const isDubMatch = /\bdub\b/i.test(match?.title || '') || /-dub$/i.test(match?.animeId || '') let best = -1 for (const rawTitle of titles) { const title = normalizeText(rawTitle) if (!title) continue let score = 0 if (baseTitle === title) score += 200 else if (baseTitle.startsWith(title)) score += 120 else if (baseTitle.includes(title)) score += 80 if (descriptorTokens.length) { score += descriptorOverlap * 50 if (descriptorOverlap === 0) score -= 120 } if (requestsDub) { if (isDubMatch) score += 40 else score -= 40 } else { if (isDubMatch) score -= 60 } if (score > best) best = score } return best }
Plain English
normalizeText() — strips punctuation and lowercases both titles so "Attack on Titan!" and "attack on titan" compare as equal.
descriptorTokens — keywords that distinguish a season or arc, like "Shippuden," "Final Season," or "Part 2." These must appear in the result too.
score += 200 — exact title match. The gold standard. A perfect hit against any title in the AniList titles array wins outright.
score += 120 — the result title starts with our search term. Common when a provider appends extra words like "Season 2."
score += 80 — our term appears somewhere inside the result. Weaker signal but still valid.
descriptorOverlap * 50 — each matching season/arc keyword adds 50 points. No overlap at all subtracts 120 — the result is probably a different arc entirely.
isDubMatch penalty -60 — if the user did not ask for a dub and this result is a dub version, subtract 60 points. Subtitles are the default preference.
return best — the function runs across every title variant (English, romaji, synonyms) and returns the highest score achieved.
🎯
The Scoring System Is Critical

"Naruto Shippuden" returns dozens of results on Gogoanime — dubbed versions, movies, wrong arcs. Without scoring, a dubbed version or the wrong season could win silently. The point system ensures the closest title with the right audio track always comes out on top.

Score Breakdown Example
+200 Exact title match — "Naruto: Shippuden" === "Naruto: Shippuden"
+120 Starts-with match — result is "Naruto Shippuden Season 2," request is "Naruto Shippuden"
+50 Per matching descriptor token — "Shippuden" appears in both request and result
-60 Dub penalty — result is a dubbed version, but user requested subtitles
-120 Zero descriptor overlap — result has no "Shippuden," likely a different arc entirely

Session-Aware Fetching

AnimePahe is a provider that requires a live browser session — it rejects requests that arrive without the right cookies. Rust acts as a persistent browser, maintaining that session across every request for the same title.

If a cookie expires mid-stream, the session manager detects the error, clears the stale session, and silently retries with a fresh one — up to three attempts.

animepahe.js async function fetchProviderTextWithSession( url, headers = {}, method = 'GET', body = null, animeId = '', preferredSessionId = null ) { let lastError = null for (let attempt = 0; attempt < 3; attempt++) { const useFreshSession = attempt > 0 const sessionId = useFreshSession ? null : resolveProviderSession(animeId, preferredSessionId) try { if (useFreshSession) clearProviderSession(animeId) const result = await invoke('fetch_anime_text_with_session', { url, headers: { ...BROWSER_HEADERS, ...headers }, method, body, sessionId, }) const text = typeof result?.text === 'string' ? result.text : '' const nextSessionId = rememberProviderSession( animeId, result?.sessionId || sessionId ) return { text, sessionId: nextSessionId } } catch (error) { lastError = error if (!isRetryableSessionError(error) || attempt >= 2) throw error } } throw lastError || new Error('AnimePahe session fetch failed') }
Plain English
animeId — the title being fetched. Used as the key to look up any existing session cookie for this anime.
attempt loop (0–2) — tries up to three times. First attempt reuses the cached session; later attempts start fresh if it expired.
useFreshSession — on retry, this is true. The old session is cleared so a brand-new cookie is obtained instead.
invoke('fetch_anime_text_with_session') — calls into Rust, which makes the actual HTTP request carrying the session cookie as a browser would.
BROWSER_HEADERS — standard browser headers (User-Agent, Accept, Referer). Without these, the server would reject the request as a bot.
rememberProviderSession() — saves the session ID returned by the server so the next request reuses it, avoiding repeated session warmup.
isRetryableSessionError() — checks if the error is a session expiry (worth retrying) versus a hard failure (not worth retrying).
🦀
Rust as the Browser

Rust's reqwest client maintains a per-provider cookie jar keyed on anime ID. It behaves like a real browser session, so AnimePahe servers accept its requests.

🔁
Three-Attempt Retry

Attempt 1 uses the cached session. If it fails with a retryable error, attempt 2 clears the cache and starts a fresh session. Attempt 3 is the final try before throwing the error up.

💾
Session Caching

After a successful fetch, the new session ID is saved to an in-memory map keyed by anime ID. The next episode request for the same show reuses the warm session instantly.

Why This Works

AnimePahe sees requests from Rust as an ordinary browser — proper headers, valid session cookie, realistic timing. The session is cached per anime ID so warm-up cost is paid only once per title. Expired sessions are silently healed with a fresh one, so the user never sees a cookie-related failure.
